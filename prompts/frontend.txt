═══════════════════════════════════════
SEMANTIC MATCHING RULES
═══════════════════════════════════════
- Analyze button text, labels, section titles to understand intent
- "Submit", "Send", "Say Hello", "Let's work Together", "Contact" near a form → POST endpoint
- Lists, grids, cards showing multiple items → GET all endpoint
- "Delete", "Remove" buttons → DELETE endpoint
- "Edit", "Update", "Save" buttons → PUT endpoint
- "View", "See", "Details" buttons → GET by id endpoint
- Never match blindly by name, always reason about context and meaning
- If no endpoint matches an element, leave it unchanged
- ONLY use endpoints from the AVAILABLE ENDPOINTS section above
- NEVER invent or create new endpoints that are not in the list
- NEVER assume an endpoint exists if it is not explicitly listed
- If a button or form has NO clear matching endpoint in the list → leave it completely unchanged
- If you are not 100% sure which endpoint to use → leave the element unchanged
- It is better to leave an element static than to connect it to the wrong endpoint
- NEVER use an endpoint just because the name is similar, the semantic meaning must match exactly
- When in doubt → do NOT connect 
═══════════════════════════════════════
FORM HANDLING RULES
═══════════════════════════════════════
- Add useState for each form field
- Add onChange handlers to each input field
- Replace static text spans used as inputs with real <input> or <textarea> elements
- Keep the exact same Tailwind classes on the input elements
- Add a handleSubmit function that calls the correct POST endpoint
- Show a success message after successful submission
- Show an error message if the request fails
- Always use try/catch for fetch calls

═══════════════════════════════════════
LIST/DATA DISPLAY RULES

- Keep the exact same component structure, just replace hardcoded props with dynamic data
- Add a loading state while fetching
