DEPENDENCY GRAPH RULES
═══════════════════════════════════════
- Identify all entities from the OpenAPI schemas
- For each entity, list which other entities it depends on
- An entity depends on another if it has a foreign key to it
- Example: Appointment has patient_id and doctor_id → depends on Patient and Doctor

Format:
{
  "EntityName": ["DependencyEntity1", "DependencyEntity2"],
  "Patient": [],
  "Doctor": [],
  "Appointment": ["Patient", "Doctor"]
}

═══════════════════════════════════════
ENDPOINT OWNERSHIP RULES (CRITICAL)
═══════════════════════════════════════
An endpoint does NOT belong to the entity in its URL prefix.
An endpoint belongs to the entity of its RESPONSE (its tag / the schema it returns).

To assign each endpoint to an entity, use this rule:
- Look at the endpoint's "tags" → the tag IS the owning entity.
- Confirm with the response schema (the $ref it returns).
- The URL prefix is IRRELEVANT for ownership.

Examples:
- GET /cuisines/{id}/recipes  → tag "Recipes", returns Recipe[]  → belongs to RECIPE
  (function list_recipes_by_cuisine goes in services/recipe.py and routes/recipe.py)
- GET /recipes/{id}/reviews   → tag "Reviews", returns Review[]  → belongs to REVIEW
- GET /authors/{id}/books     → tag "Books",   returns Book[]    → belongs to BOOK

CONSEQUENCE:
- A "list X by Y" endpoint always belongs to X (the thing returned), never Y.
- Its service function takes the parent id (e.g. cuisine_id) and queries the X table
  filtering on that foreign key. It must NEVER import or be placed in the parent entity.
- This keeps generation order valid: the parent (Y) is generated first, the child (X)
  which depends on it comes later and owns the relational endpoint.

FILE PLAN RULES
═══════════════════════════════════════
For each entity, generate exactly 4 files in this order:
1. app/models/{entity_lower}.py       → SQLAlchemy model
2. app/schemas/{entity_lower}.py      → Pydantic schemas
3. app/services/{entity_lower}.py     → business logic functions
4. app/routes/{entity_lower}.py       → FastAPI endpoints

Plus one global file at the end:
5. app/main.py                        → FastAPI app + include all routers

For each file, include:
- path: file path
- type: model | schema | service | route | main
- entity: entity name (null for main.py)
- For models: list of fields with name, type, primary_key, nullable, foreign_key
- For schemas: list of Pydantic schema classes with their fields and types
- For services: list of functions with name, input parameters, output type, description
- For routes: list of endpoints with method, path, operationId, input, output, status_code, calls_service
- For main: list of routers to include

═══════════════════════════════════════
SERVICE FUNCTION RULES
═══════════════════════════════════════
- Every service function must have db: Session as parameter
- Functions that fetch by id must raise HTTPException 404 if not found
- Use snake_case for all function names
- Function names must match the operationId from the OpenAPI spec
- Input and output types must use the exact schema names from the OpenAPI spec
- A relational list function (e.g. list_recipes_by_cuisine) belongs to the RETURNED
  entity's service (recipe), takes the parent id as int, and filters that entity's
  table by the foreign key. Never place it in the parent entity's service.

═══════════════════════════════════════
ROUTE FUNCTION RULES
═══════════════════════════════════════
- Every route must call exactly one service function
- Route function name must match operationId from OpenAPI spec
- Path parameters must be typed as int
- Response model must reference the correct Pydantic schema
- DELETE routes return status_code 204 with no response model
