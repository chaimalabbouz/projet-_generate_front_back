SEMANTIC MATCHING RULES
═══════════════════════════════════════
- Analyze button text, labels, section titles to understand intent
- "Submit", "Send", "Say Hello", "Let's work Together", "Contact" near a form → POST endpoint
- Lists, grids, cards showing multiple items → GET all endpoint
- "Delete", "Remove" buttons → DELETE endpoint
- "Edit", "Update", "Save" buttons → PUT endpoint
- "View", "See", "Details" buttons → GET by id endpoint
- Never match blindly by name, always reason about context and meaning

═══════════════════════════════════════
BUTTON CLASSIFICATION RULES
═══════════════════════════════════════

RULE 1 — UNDERSTAND THE BUTTON INTENT FIRST:
Before deciding anything, ask yourself:
"Does this button need to communicate with the backend to do its job?"
- If YES → connect to matching endpoint
- If NO → leave unchanged

RULE 2 — PURE NAVIGATION (no backend needed):
A button is pure navigation if:
- It is in a navbar/menu and its text is a page name (Home, About, Services, Blog, Portfolio, Contact page link)
- It has scrollIntoView() behavior
- Its only purpose is to move the user somewhere without any data exchange
→ Keep existing onClick unchanged, do NOT add API call

RULE 3 — BACKEND REQUIRED BEFORE NAVIGATION:
A button needs backend before navigating if:
- Its text suggests identity verification or account action (Login, Sign in, Register, Sign up, Connect)
- Its text suggests ending a session (Logout, Sign out, Disconnect)
- It is inside or near a form with credential fields (username, password, email)
→ Call the matching endpoint first
→ On success: navigate to the appropriate page
→ On error: show error message, stay on current page
→ If no matching endpoint exists in the list: leave unchanged

RULE 4 — BACKEND REQUIRED, STAY ON PAGE:
A button needs backend but no navigation if:
- It is inside a form with data fields (name, email, message, budget, subject)
- Its text suggests sending data (Submit, Send, Save, Confirm, Let's work Together, Say Hello, Contact us)
- Its text suggests modifying existing data (Update, Edit, Save changes)
- Its text suggests removing data (Delete, Remove, Cancel order)
→ Call the matching endpoint
→ Show success or error message on the same page
→ Do NOT navigate after

RULE 5 — NEVER TOUCH IF UNSURE:
If you cannot clearly classify a button into Rule 2, 3, or 4:
→ Leave it completely unchanged
→ Do not add any API call
→ Do not modify its existing onClick
